# potential tool features (for vscode plugin?)

- all-in-one tool
  - select editor variant from variant overview (angular, mail, mds, etc.)
  - variants show alerts when errors are obvious (e.g. missing keys or missing whole translation files) 

- settings
  - reference language ("de[_DE]" by default)
  - target directories for different editor components (angular, mail, etc.)
    (these should never change, but it's nice to have it)
  - list of supported languages
  - base url of an actual instance, that can show the layout context of any key that got assignes a "context url" 

- backup
  - automatically every xx minutes (keep y last backups)
  - manually
  - backup files must not clutter the data directories!

## for i18n angular files (edu-sharing-community-repository/Frontend/src/assets/i18n)

- list of categories (subdirectories)
  - probably can show/hide the list to have more space for translations 

- select active category from list of categories
  - updates viewport with keys and translations of the active category 
  - keep selection for future use

- option to toggle how keys and values are displayed
  - show as table with columns per language, first column being the key
  - show as list, grouped by key, language displayed before each value 
  - keep selection for future use

- option to toggle fields between single-line and multi-line
  - toggle all
  - toggle specific row (table view) / toggle specific value (list view)
  - in multiline, the full text must be visible without scrolling
  - keep selection for future use

- show/hide compared languages (adds/removes colums/rows to/from table/list) 
  - adding/removing a language from display must not change the viewport
    => we probably need to anchor the viewport at the top line's key
  - keep selection for future use

- add new language
  - even if a category (currently) contains no keys, there should be an empty file per language (contains only "{}")
  - do NOT fill in keys in new files, so missing keys always fall back to default language
  
- tag empty values as "REVIEW REQUIRED"

- add/edit/remove a "context hint" per key (in order to provide context for translations)

- add/edit/remove a relative "context url" per key (and/or section?) that would open a real app instance for visual context
  => uses setting "base url" as prefix
  => important for wording
  => important for available size (esp. on small elements like buttons)

- add/edit/remove keys
  - on add
    - warn if key already exists in current category
    - warn if key already exists in category 'common'
  - on update (this shouldn't happen often, so a popup to enter the new name might be okay?)
    - warn if key already exists in current category
    - warn if key already exists in category 'common'
    - prompt to 
      - update key in current category (subdirectory) only 
      - update key across all categories (subdirectories) => probably needs to list these categories and even the values?
      - cancel
  - on delete
    confirmation prompt to
    - delete from category (subdirectory)
    - delete from all categories (subdirectories)
    - cancel

- filters:
  - for keys with string/regexp (toggle ignorecase)
  - for values with string/regexp (toggle ignorecase), per language
  - for missing values

- quality checks (warnings, errors)
  - always check for warnings/errors across all files and categories, and mark affected categories and languages with error and/or warning sign
  - "group by value (of selected language)"-view, allowing to check consistency of translations of the same value 
  - a "Review" action for language/category/section
    => sets status to "REVIEW REQUIRED" on selected keys/translations in order to track review progress
    => status can be toggled to "REVIEWED" individually
    => ?? flag could automatically swith to "REVIEWED" when field is edited
  - show error about (number of) missing keys (reference: all other languages of category)
    ATTENTION: these may not be necessary, when keys are automatically added. the "error on missing translation values" would do the rest
  - show error on missing translation values
  - show error when {{templates}} don't match the one from the reference language
  - show error when {{template}} markup is in some way {broken}}
  - warn when manual file edits added translations wrong sections:
    - keys of language file A, are missing in language file B, and vice versa
    - AND the last part of the keys matches between diffs
  - toggle on/off: warn about (and filter for) keys containing lower case
      because they could be added to category "common" instead (e.g. "Unknown error") => ask torsten abt how useful this would be
  - ask torsten:
    - can keys in "common" still be overwritten and thus useful in other categories?
      if NO: warn about keys present in a specific category, when they also exist in category "common"?
      if YES: show information on keys, that they are overridden in other files (in "common") and override the "common"-entry of the same key
    - would the feature "Mark section as this-language-only" be useful?
  - keep all toggleable settings for future use

- MAYBE
  - auto-translate a value from reference language
  - button to ask chatGPT for quality check? ;)
  - mark warnings as "it's fine", similar to playwright's "expected to fail" => warning is still available, but it doesn't show as such
